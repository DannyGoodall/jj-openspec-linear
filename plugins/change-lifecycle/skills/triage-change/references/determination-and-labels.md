# Determination record + label mechanics

How `triage-change` records its verdict and applies the label. The canonical role → tracker-mechanism
map lives in `${CLAUDE_PLUGIN_ROOT}/references/triage-labels.md`; binding selection and ops live in
`${CLAUDE_PLUGIN_ROOT}/references/issue-tracking.md`. This file is the change-triage-specific detail:
the record format, the label decision rule, and how to apply the label without clobbering the categorical one.

## The determination record (posted as an issue comment)

Open with the triage disclaimer, then the structured verdict:

```markdown
> *This was generated by AI during triage.*

**Determination**
- **Type:** fix | feat | chore
- **Slug:** <kebab-name>            ← start-change reuses this for workspace/bookmark/(change-id)
- **OpenSpec:** yes | no
- **Impacted specs:** <cap>/spec.md, …   (omit if OpenSpec: no)
- **Confidence:** high | medium | low
- **Label:** ready-for-agent | ready-for-human | needs-info

**Fix sketch**
<the concrete approach — files/symbols, the shape of the change, gotchas — enough for an implementer
to run start-change and begin>
```

When `OpenSpec: no`, say *why code and specs stay in sync* (e.g. "restores the behaviour the existing
requirement already mandates"). When `OpenSpec: yes`, name each requirement that moves.

In **no-tracker mode** there is no issue to comment on: emit the same record in-thread and state that
no issue was labelled.

## Label decision rule

| Label | When |
|---|---|
| `ready-for-agent` | Fix-sketch concrete, OpenSpec call clear, **no** unresolved design choice. An AFK agent can run `start-change` and implement. |
| `ready-for-human` | A real design decision, product judgement, or risky trade-off remains. |
| `needs-info` | The fix can't be determined without more from the reporter. |

## Applying the label

Use the active issue-tracking binding (`references/issue-tracking.md`). Whichever mode, the shape is the same:

1. **Read the issue's current labels** (Linear `get_issue`; GitHub `gh issue view --json labels`).
2. **Set the desired label set:** keep the categorical label (`Bug`/`Feature`/`Improvement`), drop any
   stale state label (`needs-triage`/`needs-info`), add the new state label.
   - **Linear:** `save_issue`'s `labels` array **replaces** the whole set — so always include the labels
     to keep. For `ready-for-agent` / `ready-for-human`, **also set `state: "Todo"`** (out of Backlog);
     for `needs-info`, leave status as-is (typically `Backlog`).
   - **GitHub:** `gh issue edit --add-label <new> --remove-label <stale>`; labels are additive there.
3. **Post the determination comment** (with the disclaimer).

Linear example — a `Bug` becoming agent-ready:

```
get_issue  → labels: ["Bug"]
save_issue → labels: ["Bug", "ready-for-agent"], state: "Todo"
save_comment → the determination record above
```

Don't strip the categorical label, and don't add more than one state label at a time.

## Boundaries

- No code edits, no jj workspace, no OpenSpec proposal — those belong to `start-change` and beyond.
- **Idempotent:** if a `ready-for-human` decision is later resolved, re-run to flip it to
  `ready-for-agent` — update the *same* comment and swap the label rather than appending a new one.
